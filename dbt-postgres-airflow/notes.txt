2️⃣ psql
psql est le client en ligne de commande de PostgreSQL.
Il permet de se connecter à une base de données et d’exécuter des requêtes SQL directement depuis le terminal.

3️⃣ -U dw_user
-U indique le nom d’utilisateur PostgreSQL.
Ici, dw_user est l’utilisateur avec lequel tu te connectes à la base.

4️⃣ -d dw
-d indique le nom de la base de données à laquelle se connecter.
Ici, dw est le nom de ta base de données.

📌 dbt_project.yml  — Résumé
dbt_project.yml est le fichier principal de configuration d’un projet dbt.
👉Il sert à dire à dbt :
- le nom du projet
- quel profile utiliser pour se connecter à la base
- où se trouvent les fichiers (models, tests, macros…)
- comment créer les tables/vues (views, tables, incremental)

📌 profiles.yml — Résumé
profiles.yml est le fichier de connexion aux bases de données pour dbt.
👉 Il dit à dbt :
- à quelle base se connecter
- avec quel utilisateur
- sur quel schéma

🧠 Un warehouse = le moteur de calcul de Snowflake





2️⃣ warehouse_size = 'x-small'

👉 Taille du warehouse

x-small = le plus petit (le moins cher 💰)

suffisant pour dbt, tests, apprentissage

Autres tailles : small, medium, large, etc.

3️⃣ auto_suspend = 62

👉 Après 62 secondes d’inactivité :

le warehouse s’éteint automatiquement

🎯 Objectif :

éviter de payer quand personne ne l’utilise

📌 Bonne pratique : entre 60 et 300 secondes

4️⃣ auto_resume = TRUE

👉 Quand une requête arrive :

le warehouse se rallume tout seul

Sans ça :
❌ il faudrait le démarrer manuellement

5️⃣ initially_suspended = TRUE

👉 Juste après la création :

le warehouse reste éteint

Pourquoi ?

éviter qu’il démarre pour rien

il démarrera seulement quand une requête arrive




1️⃣ Quand utiliser VARCHAR

Quand tu veux préciser une longueur maximale pour la lisibilité ou la documentation.

Exemple classique :

name VARCHAR(50)  -- max 50 caractères
email VARCHAR(255)  -- max 255 caractères


Utile pour documents, ETL, validation de données

Bon pour SQL pur ou si tu viens d’autres bases (MySQL, PostgreSQL)

2️⃣ Quand utiliser STRING

Quand tu ne veux pas te soucier de la longueur

Utile pour les champs qui peuvent être très longs ou variables, comme description, notes, json_data…

Exemple :




| Situation                             | Type recommandé | Pourquoi                                  |
| ------------------------------------- | --------------- | ----------------------------------------- |
| Champ avec longueur connue et limitée | `VARCHAR(50)`   | Documenter et valider les données         |
| Champ très variable ou texte long     | `STRING`        | Simple, pas besoin de définir la longueur |
| dbt / pipelines modernes              | `STRING`        | Plus lisible, compatible avec Snowflake   |

	🟢 1️⃣ VIEW

Une view = une requête sauvegardée
Elle ne stocke pas les données, elle exécute le SQL à chaque fois.

👉 C’est comme un “SELECT enregistré”.

✅ Quand utiliser une VIEW ?

Tables de staging (stg_)

Transformations simples

Quand les données changent souvent

Quand tu veux économiser du stockage

Pendant le développement

Exemple :
{{ config(materialized='view') }}

Avantages :

✔ Pas de stockage
✔ Toujours à jour
✔ Rapide à créer

Inconvénient :

❌ Peut être plus lent si la requête est lourde (elle se recalcul à chaque SELECT)

🔵 2️⃣ TABLE

Une table = les données sont physiquement stockées.

👉 Le SQL s’exécute une fois, puis les résultats sont enregistrés.

✅ Quand utiliser une TABLE ?

Modèles lourds

Agrégations

Fact tables

Modèles utilisés souvent

Production

Exemple :
{{ config(materialized='table') }}

Avantages :

✔ Plus rapide à lire
✔ Stable
✔ Bon pour les dashboards

Inconvénient :

❌ Prend de l’espace
❌ Doit être reconstruite si les données changent



🔹 Conclusion
Type	Caractéristique	Ton exemple
Intermediate model	Prépare les données, pas forcément prêt pour BI	Ce que tu fais si c’est juste pour nettoyer / joindre staging
Fact model	Ligne par transaction / événement, métrique calculée	Ce que tu fais si tu veux créer la table fact_orders pour le reporting

💡 Donc selon le contexte :

Si tu prépares juste une étape intermédiaire → intermediate model

Si c’est destiné à ton fact table → fact model, et là ça correspond très bien à une fact_orders avec total_price comme métrique.

Test	Où il est défini	Comment dbt le teste
Generic / standard	YAML (schema.yml ou sources.yml)	dbt génère SQL automatiquement
Singular / métier	Fichier SQL dans tests/	dbt exécute le SQL directement

Que ce soit dans le YAML sources ou models

Ou dans les fichiers SQL sous tests/

🔹 2️⃣ À quoi sert cette base ?

Elle stocke :

📌 Les DAGs détectés

📌 Les exécutions des DAGs (dag runs)

📌 Les statuts des tâches (success, failed, running…)

📌 Les connexions (ex: Snowflake)

📌 Les variables Airflow

📌 Les utilisateurs (si auth activée)
